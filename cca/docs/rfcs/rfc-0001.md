# RFC-0001: Deterministic state-space search via `@@[persist]` snapshots

**Author:** Mark Truluck <mark.truluck@cogiton.com>
**Status:** Draft
**Created:** 2026-05-17

---

## Motivation

The existing CCA test suite (see [TESTING.md](../TESTING.md))
covers two layers well: focused mechanic tests that drive the
FSM directly, and a canonical-journey FSM-driven walk through
the real Driver. Together they catch:

- Wrong canon messages for specific verbs
- Player-visible bugs along the canonical happy path
- FSM state-machine off-by-ones on tested mechanics
- Topology disconnects

What no current layer catches: **invariant violations across the
whole reachable state space**. The canonical journey walks one
path; the focused tests check specific mechanics from teleported
start states. Neither asks "is there *any* sequence of player
commands from the start that leaves the game in an inconsistent
state?"

Concrete examples of bugs the existing layers would miss:

- A deposit-loop sequence that double-counts a treasure score
- A verb-at-specific-room combination that crashes
- An NPC state machine that, under a particular ordering of
  external events, reaches a dead-end where no transition fires
- A score value that goes negative because a hint was offered
  twice via two different paths
- A persisted-system save/restore round-trip that diverges
  *only* when the system is in a specific minority state

These are reachability bugs. Finding them requires walking the
state space — not in a hand-curated order, but exhaustively.

The cave's state machine model is rich enough to encode these,
and the `@@[persist]` infrastructure on Adventure (and all its
composed sub-systems) already supports the operation the search
needs: take a complete snapshot of the world, try an action, and
roll back to try a different action.

## The contract

The key words **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**,
**MAY** are to be interpreted as in RFC 2119.

### The search

A new test scaffold — `tests/test_cca_state_space.gd` — does
breadth-first search over the reachable state space starting
from a known initial state (a fresh `Cca.new()` with default
aspects, or any persisted save). At each visited state the
harness:

1. Takes a snapshot via `adv.save_state()` — a `PackedByteArray`.
2. Computes a **canonical hash** of the state (see below).
3. If the hash has been visited before, prunes.
4. Otherwise, enumerates the legal **actions** at this state.
5. For each action:
   a. Restores the snapshot from step 1.
   b. Issues the action through the real Driver.
   c. Asserts every invariant (see below).
   d. Pushes the resulting state onto the BFS queue.

The search terminates when the queue is empty (all reachable
states visited) or a depth/time limit is hit.

### Canonical state hash

State distinguishability is a deliberate, configurable design
choice. The default hash includes:

| Dimension | Source | Notes |
|---|---|---|
| `player_room` | `adv.player_room()` | Always significant |
| Inventory (sorted) | `[id for id in CARRYABLE_IDS if adv.player.carrying(id)]` | Order-insensitive |
| NPC states | `[bird.get_state(), snake.get_state(), bear.get_state(), troll.get_state(), pirate.get_state()] + [dwarfN.get_state() for N in 1..5]` | Concatenated strings |
| Key-item states | `lamp.get_state()`, `bottle.contents()`, `plant.size()`, `grate.locked()`, `bridge.built()`, ... | One per puzzle FSM |
| Endgame phase | `adv.endgame_state()` | "active" / "closing" / "in_repository" / "won" |

What the default hash explicitly **excludes**:

- `score` — informational only; asserted as an invariant, not
  used for dedup
- `turn_counter` / `lamp.battery` / hint streak counters — every
  step would otherwise be a new state and the search wouldn't
  terminate
- Position of stalking dwarves and pirate — these walk randomly
  per turn; including them blows up the search

A harness option SHOULD allow the user to widen the hash (e.g.
include dwarf positions for a stricter search) at the cost of
runtime.

### Action enumeration

At each visited state the harness derives a list of actions to
try. The action list is the union of:

1. **Direction commands** — every key in `room_exits[current_room]`.
2. **Magic-word teleports** — `xyzzy` / `plugh` / `plover` if
   the current room is one of their canonical sources.
3. **Object interactions** — for each item `is_in_room(current)`
   AND each item `player.carrying(id)`, the canonical verbs:
   `take`, `drop`, `examine`, `read`, plus item-specific verbs
   (`light`/`extinguish` for the lamp, `fill`/`empty`/`pour` for
   the bottle, etc.).
4. **NPC interactions** — `attack`, `feed`, `release` if the
   relevant NPC is in the current room.
5. **UI verbs** that affect game state — `save`, `load`, `score`,
   `inventory` (these are no-ops for dedup but should still be
   verified to not crash).

Verbs that don't change state OR don't affect any tracked
dimension MAY be omitted from the enumeration (`look`, `help`,
`info`) — they only waste search time.

### Invariants

Every state transition asserts a fixed set of invariants. A
violation halts the search with the offending command sequence
recorded.

The minimum-viable invariant set:

- `0 ≤ adv.score()` — score never goes negative.
- `0 ≤ adv.lamp.battery_left() ≤ adv.lamp.MAX_BATTERY`
- `1 ≤ adv.player_room() ≤ 140` — player room is always valid.
- `adv.player.inventory_size() ≤ BACKPACK_LIMIT` — no overflow.
- `endgame_state` transitions monotonically through `active →
  closing → in_repository → won` (or `→ permadead`); never
  jumps backwards or skips states.
- Save/restore round-trip: at every visited state, `save_state()`
  → `restore_state()` produces a state whose canonical hash
  matches the original.
- Carrying-an-item ⟺ that item's `_item` system is in its
  `$Carried` state (no inconsistencies between Player and Item
  views of inventory).

Additional invariants are easy to add as more bug-classes are
identified. The set above is the seed.

### Determinism

Probabilistic events (Witt's End 95/5, Bedquilt random walk,
dark-pit-fall 35%, dwarf walks, pirate steals) **MUST** be
controllable for the search to be reproducible. See [the
Determinism section below](#determinism-prerequisite) for the
prerequisite work.

When deterministic mode is enabled, every action at every state
produces the same outcome, every run. The reachability graph is
a function of `(rng_seed, initial_state, action_set)`; tests can
fail with a precise reproducer ("seed=42, walk these commands
from the start, observe the invariant violation").

## Examples

### Bug class — score-double-count

The current canonical journey deposits gold once (canonical
walk). An exhaustive search would also try every other path
that reaches "gold in inventory, player at room 3" — including
unusual routes that go via the secret-canyon shortcut. If any
of those routes credits the gold score twice (a real bug class
on persist-restore systems), the invariant `treasure_score ==
sum(deposited_treasures.value)` fails and the harness reports
the exact action sequence.

### Bug class — NPC dead-end

The pirate FSM has $Dormant → $Stalking → $Vanished. If, under
a specific external-event ordering, the pirate transitions
to a state with no outbound events at the wrong moment, the
canonical journey wouldn't notice. The exhaustive search exercises
every reachable (pirate state × player room × treasures held)
combination, so a dead-end surfaces as a state from which no
action enumerates.

### Bug class — save/restore divergence in minority state

The save-state round-trip tests in Layer 1 check restore from a
fresh state and after a few specific commands. An exhaustive
search round-trips at every visited state — including ones with
unusual combinations like "carrying chain + bear is $Released +
troll is $Vanished + at room 122 with bridge intact." If any
domain field doesn't round-trip cleanly, the canonical-hash
comparison fails and the offending state is reported.

## Determinism — prerequisite

The search relies on actions producing reproducible outcomes.
Today, three categories of CCA non-determinism break that:

### 1. Per-NPC seeded PRNG (already deterministic)

`Dwarf(seed: int)` and `Pirate(seed: int = 42)` carry their own
RNG state baked into `@@[persist]`. Seeds are set at composition
time; their walks are reproducible. **No work required.**

### 2. Driver-side probability rolls (NOT seeded today)

`driver.gd` calls bare `randi()` for:

- Dark-pit fall (35% per move in a dark room)
- Y2 hollow voice (25% per visit to canon room 33)
- Forest disorientation
- Bedquilt random-walk bumpers (80% / 50% chained)
- Witt's End 95/5 bounce
- The unknown-verb-msg #60 vs #61 vs #13 distribution

The fix is mechanical: thread a `RandomNumberGenerator` instance
through the driver; replace every bare `randi()` with
`rng.randi()`. Production wires `RandomNumberGenerator.new()`
with `randomize()`; tests inject a seeded one.

### 3. FSM-side bumper rolls

The `gated_exits` table in `topology.gd` carries
`{"check": "probability", "pct": N}` rules that are resolved
inside the driver — fall into category (2). No FSM-internal
non-determinism (other than the per-NPC PRNG already covered)
is known.

The state-space search **MUST NOT** ship until all three
categories are routed through a controllable RNG. Estimated
work: ~10-15 callsite changes in `driver.gd`, one new domain
field on the driver, ~3 hours of focused work.

## Search-runtime estimate

Back-of-envelope for a default-hash search of the full canon:

```
~140 rooms × ~5 inventory configs × ~8 NPC-state configs
  ≈ 5,600 nodes (rough upper bound; many combinations unreachable)

Each node: ~30 actions enumerated × (save + restore + invariant
checks) ≈ ~50ms per action

Total: 5,600 × 30 × 50ms ≈ 8 minutes
```

Acceptable for a nightly / pre-merge gate, not for the per-
commit suite. The fast `run_tests.sh` stays focused on Layer 1
+ canonical journey; the state-space search is its own
`tests/test_cca_state_space.gd` with its own timeout (10 min)
and its own `--state-space` switch on `run_tests.sh` for
optional inclusion.

## Alternatives

### A. Hand-curated multi-path coverage

Write 20-50 additional canonical-journey-style FSMs that each
walk a different path. Catches some reachability bugs the
single journey misses.

**Rejected.** Doesn't scale — every new path is hand-authored.
Misses the bugs that surface only in combinations the author
didn't think to write.

### B. Property-based testing (Hypothesis-style)

Random-generate action sequences, run them, assert invariants.
Catches some reachability bugs probabilistically.

**Partial fit, not a replacement.** Random sequences exercise a
fraction of the state space and don't terminate at "all reachable
states visited." Useful as a complement (deeper coverage of
common-case action sequences) but doesn't replace exhaustive
search for proving invariants hold.

### C. Real model-checker (SPIN, TLA+, etc.)

Translate the CCA FSM to a model-checking language; verify
properties symbolically.

**Rejected.** Translation cost is high (CCA's state is encoded
in 26 composed FSMs with intricate cross-FSM coordination);
maintaining the model in lockstep with the Frame source is
ongoing work that adds friction to changes. The proposed
search reuses the live Frame state machine directly — no
model-translation step.

### D. Skip determinism, search with probabilistic actions

Run the search without seeding RNG; accept that a node visit
might produce different child states across runs.

**Rejected.** Breaks reproducibility. A failing assertion needs
a precise reproducer; non-deterministic search produces
"sometimes this fails, can't tell you when." Determinism is
prerequisite, not optional.

## Migration

No source-level migration. The work splits into two phases:

**Phase A — determinism prerequisite.**

- Thread `RandomNumberGenerator` through `driver.gd` and
  `cca_main.gd`. Replace bare `randi()` with `rng.randi()`.
- New domain field `rng: RandomNumberGenerator` on the Driver
  Control.
- `_test_helpers.gd:make_driver()` accepts an optional seed.
- Existing tests unaffected (default seed is `randomize()`).

**Phase B — state-space harness.**

- `tests/test_cca_state_space.gd` — BFS over reachable states.
- `cca/scripts/state_space.gd` — reusable harness (state hash,
  action enumeration, invariant set).
- `run_tests.sh --state-space` flag to opt in (excluded from
  the default per-commit run).
- Documentation in `TESTING.md` ("Layer 4 — exhaustive search").

## Drawbacks

- **Runtime.** Even with state-hash dedup, an 8-minute search
  is too slow for the per-commit hook. Lives in nightly /
  pre-merge gates, not in the fast loop.
- **False reachability.** The search visits states that aren't
  reachable in canonical play (e.g. "carrying chain + at room
  1" via an unusual path). Some invariant violations the search
  flags might be canonically unreachable; the report should
  include the action sequence for triage.
- **State-hash blind spots.** Anything excluded from the hash
  (turn counter, dwarf positions, score) is invisible to
  dedup; bugs that surface only in those dimensions need a
  widened hash to find. The default hash is a starting point,
  not the answer.
- **Maintenance.** Every new `@@system` added to Adventure's
  domain may need a hash-function update. Without that update,
  the search treats different states as equivalent and may
  miss bugs OR loop. Either auto-derive the hash from
  `@@[persist]` state, or document the per-system update.

## Unresolved questions

- **Hash auto-derivation.** Can the hash function be derived
  mechanically from each persisted system's domain fields,
  rather than hand-curated? If yes, the per-system maintenance
  cost goes away. If no, the curated list is the answer.

- **Action timeout policy.** Some actions (`save`, `load`)
  take noticeably longer than others. Should the harness time
  out an individual action and report, or always wait?

- **Initial-state seeding.** Should the search start from
  `Cca.new()` only, or also from a corpus of "interesting" save
  files (mid-game checkpoints)? The latter widens coverage at
  the cost of running the search N times.

- **Invariant set growth.** What's the right pace for adding
  invariants? Add one per discovered bug? Add aggressively
  upfront? The minimum-viable set above is conservative.

- **Naming.** "State-space search" vs "exhaustive search" vs
  "model check" — what's the canonical name in CCA's docs?
  The implementation isn't a real model checker (no symbolic
  reasoning), so the third term overpromises.
