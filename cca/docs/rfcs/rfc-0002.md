# RFC-0002: Multi-canonical-paths as testing spine + Go-Explore demonstration seeds

**Author:** Mark Truluck <mark.truluck@cogiton.com>
**Status:** Draft
**Created:** 2026-05-18

---

## Motivation

The existing CCA test suite has accumulated a strong but
fragmented testing stack:

- 60+ FSM-direct unit tests, each scenario set up via teleport
  + force-take.
- 1 canonical-journey FSM-driven end-to-end test
  ([canonical_journey.fgd](../../frame/canonical_journey.fgd)),
  walking $AtRoad → $Victory through the real Driver.
- Phase A state-space BFS over three teleport-based sweeps
  ([rfc-0001.md](rfc-0001.md)).
- Phase B coverage probe (LFU + Go-Explore archive + BFS-routed
  action storm).
- Phase C model-based testing — item placement, NPC anchoring,
  treasure values, verb-effects (14 entries today).

The canonical journey is **isolated**. Nothing else references
its milestone states. Each unit test, each spec entry, each
probe walk reconstructs setup state from scratch via the
generic `setup_steps` / teleport + force-take pattern. This
creates three problems:

1. **Setup brittleness.** Tests that need a mid-game world
   state (e.g. bear taming after grate descent, or bridge-built
   for endgame approach) have to either manually drive a long
   sequence of commands or use FSM-state injectors that bypass
   canon semantics. The Phase C `bear_on_troll_bridge_collapses`
   verb-effect was abandoned for exactly this reason — the canonical
   setup chain was too tangled to encode declaratively.

2. **Probe seeding gap.** The Phase B probe spends most of its
   per-walk budget rediscovering the canonical-start → past-grate
   → past-snake → ... prefix. Real coverage gains come from
   exploring around the *deep* states — past the grate, past the
   snake, past the dragon, in the endgame. With no seed
   mechanism for those deep states, the probe statistically
   under-visits them.

3. **No targeted-coverage closure mechanism.** When the probe
   reports a (room, action) cell at count 0, there is no clean
   way to write "a focused canonical path that walks through
   this cell" and have it integrated into the test infrastructure.

The Go-Explore literature (Ecoffet et al. 2019/2021) names the
fix for problem 2 explicitly: **expert demonstrations as
archive seeds**. Phase 1.5 of Go-Explore takes human or scripted
demonstrations, snapshots their state trajectory, seeds the
archive, and lets random exploration concentrate around the
demonstrations. This is the trick that makes Go-Explore solve
hard-exploration games like Pitfall that random alone cannot
crack.

We have something better than human demonstrations: **the FSM
itself is the demonstration script.** The canonical journey is
already a Frame state machine that produces a known-good
trajectory. The work to capture milestone snapshots and feed
them to the probe archive is straightforward.

This RFC proposes the architectural shift.

## The contract

The key words **MUST**, **MUST NOT**, **SHOULD**, **SHOULD NOT**,
and **MAY** are used per RFC 2119.

### Multi-path catalog

There **MUST** be a `cca/frame/canonical_paths/` directory
containing multiple journey FSMs. Each journey:

- **MUST** be a Frame state machine declaring an ordered sequence
  of milestone states, each with `commands_from_previous`,
  `expected_room`, and prose-log expectations.
- **MUST** be drivable end-to-end through `Driver._process_input`
  (no FSM-direct shortcuts permitted in the canonical journey
  itself — the whole point is canon fidelity through the real
  parser pipeline).
- **MUST** terminate in a defined state (not necessarily $Victory;
  a journey that ends at "treasures all deposited, ready for
  closing" is valid).
- **SHOULD** be named for the segment of the game it exercises
  (`canonical_journey.fgd` is the happy path; variations like
  `canonical_journey_dragon_first.fgd`,
  `canonical_journey_witts.fgd`,
  `canonical_journey_maze_deep.fgd` are encouraged).

### Milestone snapshot export

Each journey's harness **MUST** capture `fsm.save_state()` bytes
at each declared milestone state and expose them via a
`MilestoneRegistry` global:

```gdscript
MilestoneRegistry.get_snapshot("canonical_journey", "AtBridge")
MilestoneRegistry.get_snapshot("canonical_journey_dragon_first", "DragonDead")
```

Snapshots **MAY** be precomputed once per test-suite run and
cached; canonical paths are deterministic given a fixed RNG seed,
so cache validity is verifiable.

### Probe seeding from milestones

The Phase B probe **MUST** accept a `seed_journeys: Array[String]`
config field. When non-empty:

- Each journey is run-once at probe startup to populate the
  archive with all its milestone snapshots.
- The Go-Explore "Go" phase samples from milestone snapshots in
  addition to probe-discovered cells.
- Sampling weight **SHOULD** favor milestones from journeys that
  reach territory the probe is statistically unlikely to find
  unaided (deep cave, endgame).

### Phase C verb-effect spec migration

The `world_spec.gd` `VERB_EFFECTS` array **SHOULD** offer a
`milestone` setup field that loads a named snapshot in lieu of
the explicit `setup_steps` chain:

```gdscript
{
    "id":        "bear_on_troll_bridge_collapses",
    "setup":     {"milestone": "canonical_journey:BearChamberWithChain"},
    "input":     ["over"],
    "expect":    {"player_state": "dead",
                  "prose_includes": ["bridge", "collapse"]},
},
```

The `setup_steps` mechanism stays in place — the new field is
additive, used when the milestone is the cleanest representation.

### Coverage gap closure workflow

For under-covered (room, action) cells identified by probe
reports, the workflow **SHOULD** be:

1. Probe runs, emits report listing cells at count 0.
2. For each gap, identify the shortest canonical path that
   routes through the gap.
3. Add a new journey (or extend an existing one) that walks
   that path.
4. Re-run the probe with the new journey in `seed_journeys`.
   The probe's coverage now includes the previously-uncovered
   cell.

This closes the loop: probe finds gaps, canonical paths fill
them, probe re-verifies. The gap-closure is *content-additive*
(more paths = more coverage) rather than algorithmic, which
makes it incrementally bankable.

## Examples

### Example 1: Bear-bridge-collapse spec entry

Before (today, abandoned during Phase C):
```
"setup": [
    {"goto": 3}, {"cmd": "take food"},
    {"goto": 8}, {"cmd": "take keys"},
    {"goto": 8}, {"cmd": "unlock grate"},
    ... 15 more steps including troll-bridge payment ...
    {"goto": 130}, {"cmd": "feed bear"},
    {"goto": 130}, {"cmd": "take chain"},
    {"goto": 117},
],
"input": ["over"],
```

After:
```
"setup": {"milestone": "canonical_journey:BearChamberWithChain"},
"input": ["over"],
```

### Example 2: Probe seeded with multiple journeys

```
var probe = Probe.new()
probe.seed_journeys = [
    "canonical_journey",           # happy path
    "canonical_journey_dragon_first",
    "canonical_journey_maze_deep",
    "canonical_journey_witts",
]
probe.run()
```

Each journey runs once; their milestone snapshots populate the
archive. Subsequent probe walks sample from a mixed pool that
includes deep-cave states the probe alone wouldn't reach.

### Example 3: Targeted gap closure

Probe report shows `127:examine spices` at count 0 across 50
walks. Diagnosis: the spices treasure room (canon 127) is only
reachable via a specific bedquilt route, and the probe's
random-walk-from-canonical-start hits it < 5% of the time.

Fix: add `canonical_journey_spices.fgd` — a 12-state path that
walks from canonical start to canon 127 with EXAMINE SPICES as
the terminal milestone. Add to `seed_journeys`. Probe re-runs;
`127:examine spices` is now at count ≥ 1 from the seeded archive
state plus follow-on walks.

## Determinism prerequisite

Multi-canonical-paths inherits RFC-0001's RandomNumberGenerator
threading. Each canonical journey **MUST** be deterministic
under a fixed seed — a path that produces different milestone
snapshots run-to-run defeats the seeding architecture. The
existing canonical_journey already meets this; new variants must
preserve the property.

## Search-runtime estimate

Per-journey replay through the Driver is roughly 200 commands
× ~1ms = 200ms. Running 8 journeys at startup adds 1–2 seconds
to a probe run. Negligible compared to the per-walk explore
budget.

Memory: 8 journeys × ~30 milestones each = 240 snapshots × ~2KB
per save_state = 480KB. Well within reasonable.

## Alternatives considered

**A. Hand-coded setup helpers per spec entry.** What we have
today. Doesn't scale; bear-bridge-collapse spec entry was
abandoned because the setup was too tangled.

**B. FSM-state injectors (e.g. `fsm.bear.force_to_following()`).**
Bypasses canon semantics. A test that uses an FSM-injector to
reach state X doesn't prove the player can canonically reach X;
it only proves the FSM behaves correctly *given* X. The
canonical-paths approach proves both.

**C. Record-and-replay of UI sessions.** Capture actual playthroughs
as command logs, replay them. Brittle (canon prose changes break
replays); requires a recording mechanism; doesn't scale to many
variants.

The multi-canonical-paths approach combines the strengths of
each: structured like FSMs (A's expressivity), canon-correct
like real play (B's bypass-free path), and replayable like
sessions (C's repeatability) — minus the brittleness.

## Migration

This RFC's deliverables are additive. No existing test changes
shape. Migration is:

1. Create `cca/frame/canonical_paths/` and move
   `canonical_journey.fgd` into it (preserving the existing test
   binding).
2. Add `MilestoneRegistry` global + snapshot-export harness.
3. Extend the probe to accept `seed_journeys`.
4. Add 2–4 alternative journeys covering under-probed sectors
   (dragon-first, maze-deep, witts, plover-route).
5. Migrate any Phase C verb-effects entries that would benefit
   from milestone setup (notably the abandoned bear-bridge
   entry).
6. Document the gap-closure workflow in TESTING.md.

Phases 1–3 are the foundational pieces; 4–6 are content.

## Drawbacks

- **Journey-maintenance burden.** Every canonical path becomes a
  test that has to be maintained when canon mechanics change.
  Mitigated by the existing canonical_journey already paying this
  cost — adding 4 more variants is incremental.

- **State-snapshot size growth.** As journeys multiply, the
  cached snapshots accumulate. At 8 journeys × 30 milestones the
  cache is ~500KB; at 20 journeys × 30 it's ~1.2MB. Still small.

- **Drift between journeys and canon.** If a journey's mid-game
  command sequence becomes invalid because of an unrelated
  mechanic change, the journey test breaks. This is a feature,
  not a bug — that's the canon-drift alarm. But it does mean
  more tests to keep green.

## Unresolved questions

- Should milestone snapshots include the Driver's state (input
  history, RNG seed, etc.) or only the FSM's? The existing
  `save_state` mechanism captures only the FSM; the Driver's
  RNG and input-history aren't persisted. For probe-seeding the
  FSM-only snapshot is enough; for canonical-replay tests the
  Driver state may matter. Defer until a concrete use case
  emerges.

- Should journeys be allowed to *branch*? E.g. one journey for
  `BearChamber` that splits into `BearTamed` and `BearAttacked`
  successor states. Current canonical_journey is linear. Branching
  is more expressive but more complex. Defer to a follow-on RFC
  if the linear-only model proves limiting.
