# advent.dat — full section-by-section inventory

Source file: `cca/canon/advent.dat` (1808 lines, ASCII, tab-delimited).
Decoder spec: `cca/canon/advent.for` lines 95-195 (FORTRAN comment block).

Each section in `advent.dat` is wrapped:
```
<section-number>     ← single int on its own line
<row>                ← tab-delimited fields
<row>
...
-1                   ← end-of-section marker
```
Section 0 on the final line is the database terminator. There is no
preamble — the file opens with `1\n` (the section-1 header).

Section numbering matches the FORTRAN spec. The user's request used
'section 2 = travel table' — that is FORTRAN section 3. This document
uses FORTRAN numbering throughout. The audit script at
`/tmp/conditional_audit.py` counts `-1` boundaries (so its
`section_count==1` = FORTRAN section 3 = travel table).

Section line ranges (1-indexed in advent.dat — header line is
not counted as data; boundary `-1` line is also not data):
- Section  1 — long descriptions:    header  1, body    2- 301 ( 300 rows), boundary 302 `-1\tEND`
- Section  2 — short descriptions:   header  303, body  304- 368 (  65 rows), boundary 369 `-1`
- Section  3 — travel table:         header  370, body  371- 863 ( 493 rows), boundary 864 `-1`
- Section  4 — vocabulary:           header  865, body  866-1160 ( 295 rows), boundary 1161 `-1`
- Section  5 — object descriptions:  header 1162, body 1163-1311 ( 149 rows), boundary 1312 `-1`
- Section  6 — arbitrary messages:   header 1313, body 1314-1640 ( 327 rows), boundary 1641 `-1`
- Section  7 — object locations:     header 1642, body 1643-1697 (  55 rows), boundary 1698 `-1`

- Section  9 — COND bits:            header 1732, body 1733-1745 (  13 rows), boundary 1746 `-1`
- Section 10 — class messages:       header 1747, body 1748-1756 (   9 rows), boundary 1757 `-1`
- Section 11 — hints:                header 1758, body 1759-1766 (   8 rows), boundary 1767 `-1`
- Section 12 — magic messages:       header 1768, body 1769-1806 (  38 rows), boundary 1807 `-1`
- Section  0 — terminator:           line   1808

**Format quirk**: the boundary terminating section 1 is `-1\tEND`
(line 302) instead of the bare `-1` used by all later sections. The
FORTRAN parser at `advent.for` line 245 reads `IF(LOC.EQ.-1)GOTO 1002`
via `FORMAT(1G,15A5)` — the `\tEND` after the `-1` is just consumed by
the 15A5 trailing field and ignored. A naive parser that does
`line.strip() == '-1'` will miss the section-1/2 boundary entirely and
merge sections 1+2 into one big block. The decoder used here splits on
`parts[0] == '-1'` to handle this.

- `M == 0`     → unconditional

- `500<M<=600` → prop(M mod 100) must NOT be 2

with a *different* destination value but the same room/verb. Verbs only

## Magic words — note on the user's 'section 13'

The user's request mentions a *section 13: magic words (canonical word
→ action)*.  The 1977 canon `advent.dat` does **NOT** have a section 13.
Magic words are encoded as **motion verbs** in section 4 (M=0) — see the
section-4 dump above. Specifically, the words XYZZY (1), PLUGH (65/71),
PLOVER, FEE/FIE/FOE/FOO/FUM appear as either motion verbs or action verbs
there. The canonical word→action wiring lives in `advent.for` itself
(routines 30100-30303 plus the verb dispatch table). The data file only
stores the dictionary entry.

**Surprises / judgment calls**:

1. The user's request lists 13 sections, treating 'section 2' as the
   travel table; I used the FORTRAN spec's 12-section numbering throughout.
   The user's '/tmp/conditional_audit.py' counts `-1` boundaries (so its
   `section_count==1` already corresponds to FORTRAN section 3).
2. There is no section 13 'magic words' table — magic words are vocab
   entries in section 4 (mostly motion verbs) and the dispatch logic
   lives in advent.for routines 301/302/303.
3. Section 5 prop messages can be `>$<`, which is a placeholder meaning
   'no message for this prop value' — flagged in the dump.
4. Section 9 lines can wrap across multiple physical rows for the same
   bit (rooms continue on the next line). The decoder concatenates them.
5. Section 11 hint #1 is unused in canon (bit 0 is LIT, not a hint).
   Hints #2 and #3 are repurposed as engine state flags (per the FORTRAN
   comment at line 187-191) — their cost/turn fields drive different
   subsystems. Real player-facing hints are #4..#9.
6. The travel table contains some unconditional rows mixed in with
   conditional ones; the per-room exit summary deliberately keeps source
   order so the fallthrough chain is readable.
